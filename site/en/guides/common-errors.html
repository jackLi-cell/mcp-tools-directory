MCP Server Common Error Troubleshooting

This article summarizes the most common error messages and solutions encountered during MCP Server configuration and usage. When facing issues, start by checking client logs for specific error messages.

How to View Error Logs

The first step in troubleshooting is finding the log files:

  • Claude Desktop (macOS): ~/Library/Logs/Claude/mcp*.log
  • Claude Desktop (Windows): %APPDATA%\Claude\logs\mcp*.log
  • Cursor: Click the Server name in MCP settings page to view error details

Error 1: command not found

Error message: spawn npx ENOENT or command not found: npx

Cause: The client cannot find the specified command when starting the Server.

Solution:

  1. Confirm the command is installed: run which npx (macOS/Linux) or where npx (Windows) in terminal
  2. If using nvm/fnm or other version managers, use the full path in config: 
    "command": "/Users/yourname/.nvm/versions/node/v20.0.0/bin/npx"
  3. For uvx, confirm uv is installed: curl -LsSf https://astral.sh/uv/install.sh | sh

Error 2: Connection Timeout / Server Not Responding

Error message: MCP server connection timeout or Server status keeps showing "connecting"

Cause: The Server process started but did not correctly respond to the MCP protocol handshake.

Solution:

  1. Manually run command + args in terminal to confirm the command executes normally
  2. Check if interactive prompts are blocking startup (npx missing the -y parameter)
  3. Confirm network connection is working (some Servers need to download dependencies on startup)
  4. Check if a firewall or proxy is blocking local communication

Error 3: JSON Config Parse Failure

Error message: SyntaxError: Unexpected token or client cannot read config

Cause: The config file is not valid JSON format.

Solution:

  1. Check for trailing commas (no comma after the last element)
  2. Confirm all key names are wrapped in double quotes
  3. Use a JSON validation tool: cat config.json | python -m json.tool
  4. Note that Windows paths with backslashes need to be escaped as \\

Error 4: Permission Denied

Error message: EACCES: permission denied

Cause: The Server is trying to access a file or directory without permission.

Solution:

  1. Confirm the paths specified in args exist and the current user has read permission
  2. Do not point to system-protected directories (e.g., /System, C:\Windows)
  3. On macOS, you may need to grant access in "System Settings > Privacy & Security > Files and Folders"

Error 5: Environment Variable Not Set

Error message: API key not found or Missing required environment variable

Cause: The API Key or Token required by the Server is not configured in the env field.

Solution:

  1. Check the Server documentation to confirm which environment variables are needed
  2. Add the corresponding key-value pairs in the config's env field
  3. Confirm the key value is correct and not expired
"env": {
  "GITHUB_TOKEN": "ghp_your_actual_token_here"
}

Error 6: Version Incompatibility

Error message: Unsupported protocol version or unexpected behavior

Cause: The MCP protocol version between client and Server does not match.

Solution:

  1. Update the client to the latest version
  2. Update the Server package: clear cache and re-run npx (npx will automatically pull the latest version)
  3. Check the Server repository README to confirm supported protocol versions

General Troubleshooting Steps

  1. Check log files for specific error messages
  2. Manually run command + args in terminal to confirm the command is executable
  3. Use a JSON validation tool to check config file format
  4. Fully restart the client (not just refresh the window)
  5. Try a minimal config (keep only one Server) to rule out conflicts

FAQ

What if the problem persists after restarting?
Try removing the problematic Server from the config file and keep only one known working Server for testing. If a single Server also fails, it may be a client version issue - try updating or reinstalling the client.
What if logs contain no useful information?
Manually run the Server command in terminal and observe the output. For example: npx -y @modelcontextprotocol/server-filesystem /tmp, and check for errors.
How to permanently fix npx path issues on macOS?
Find the full path of npx (which npx), then use the full path in the config instead of "npx". If using nvm, the path is usually ~/.nvm/versions/node/vXX.X.X/bin/npx.
